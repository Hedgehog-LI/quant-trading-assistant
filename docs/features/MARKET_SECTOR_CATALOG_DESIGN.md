# 市场板块目录、关注与快照设计

> P1.6 自动采集、全市场历史榜单、频率和可靠性设计见
> `MARKET_SECTOR_AUTOMATIC_COLLECTION_DESIGN.md`。本文保留 P1.5 目录发现与手工快照基线。

## 1. 目标与边界

板块能力分成三个互不混用的对象：

- **市场板块目录**：Longbridge 返回的 A/H/US 行业排行、层级和成分行情。
- **市场行业关注**：用户从目录中明确选择的行业，保存 provider 稳定标识和可选 ETF/指数代码。
- **自定义分组**：用户手工维护的个人观察池，继续使用 `market_segment` 两表。

系统只做只读行情采集和分析，不自动交易。板块聚合资金净流入来自 provider 成分字段求和，必须与采集时间、延迟标志一起解释。

## 2. 已交付范围（P1.5）

1. CN/HK/US 行业排行与层级查询。
2. 行业成分查询，包含现价、涨跌、成交量、成交额、净流入、股本、标签和延迟标志。
3. 从排行关注行业；可保存一个关联 ETF/指数统一证券代码。
4. 手动采集一份不可变行业聚合快照和成分快照。
5. 关注列表、启停、删除、历史快照和最新成分查询。
6. 前端“市场板块 / 我的关注 / 自定义分组”完整操作面。

## 3. Provider 实现决策

官方 Java SDK 4.3.3 的 Java 类声明了行业方法，但发布包 native 未导出对应 JNI 符号。行业能力因此不走反射 JNI，而由 `LongPortIndustryHttpClient` 按官方同版本签名规则调用只读 HTTPS：

- `GET /v1/quote/industry/rank`
- `GET /v1/quote/industries/peers`
- `GET /v1/quote/index-constituents`

认证信息仍只从部署环境读取，不进入数据库、前端、Git 或日志。证券报价/K 线继续使用原有 SDK adapter，两条边界互不影响。

## 4. 数据模型

| 表 | 责任 | 关键约束 |
| --- | --- | --- |
| `market_sector_watch` | provider 行业身份、市场、名称、可选跟踪证券、启停和最后结果 | `provider_code + provider_sector_id` 唯一 |
| `market_sector_snapshot` | 某次采集的行业涨跌、年内涨跌、龙头、宽度、资金和成交聚合 | P1.6 按 `watch_id + snapshot_bucket_time` 保证自动采集时间桶幂等；手工快照桶可空 |
| `market_sector_member_snapshot` | 某次采集的成分证券行情和资金事实 | `snapshot_id + canonical_symbol` 唯一；随快照级联删除 |

provider 行业 ID 是外部稳定身份，示例：A 股 `BK/SH/IN40159`、港股 `BK/HK/...`、美股 `BK/US/...`。A 股不能假设为 `BK/CN/...`。

## 5. 刷新流程

1. 根据 `market + providerSectorId` 查询层级并校验身份。
2. 查询行业成分；过滤无法转换为统一证券代码的异常项。
3. 计算成分净流入、成交额、成交量合计，并以涨幅最高成分作为当次领涨标的。
4. 在单个事务中写入聚合快照、全部成分快照并更新关注水位。
5. 任一步失败不写半份快照，关注记录保留上次成功时间并记录本次错误。

创建关注会立即执行一次该流程。后续由用户点击“采集”手动刷新；当前不默认开启全行业定时轮询，避免消耗 provider 配额和在非交易时段制造重复快照。

## 6. ETF/指数联动

`tracking_symbol` 只保存关联关系，不伪造 ETF 等同于行业。需要连续 K 线时，用户使用该统一证券代码复用行情工作台现有日 K/分钟 K 采集计划。A 股 `5xxxxx` ETF 识别为上交所证券。

## 7. 验收事实

- 真实只读最小调用：CN/HK/US 行业排行成功；CN 行业层级和成分资金字段成功。
- 后端 284 tests 通过，Flyway V14 在 MySQL 兼容测试中迁移成功。
- 前端 typecheck、lint 和全量测试通过；Docker/curl/浏览器结果以验收日志最新条目为准。

## 8. 后续分析层

在本轮原始事实稳定后，再基于历史快照派生板块相对强弱、资金趋势、龙头贡献度、宽度变化与异动提醒。派生指标必须记录公式、窗口、样本覆盖和失效场景，不覆盖原始快照。
