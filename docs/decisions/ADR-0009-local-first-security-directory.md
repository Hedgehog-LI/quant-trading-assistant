# ADR-0009：证券名称检索采用本地目录优先、外部来源补全

- 状态：Accepted
- 日期：2026-07-17

## 背景

多个页面要求用户手工填写统一证券代码。实时调用单一行情 provider 做名称联想，会让输入体验受网络、额度、权限和 provider 可用性影响；LongPort Static Info 等接口也通常要求调用方已经知道证券代码。另一方面，把所有市场价格历史全量落库成本过高，与个人系统的按需数据资产目标不符。

## 决策

1. 扩展现有 `stock_basic`，建设 A/H/US 本地证券元数据目录；不新建含义重复的证券主表。
2. 新增别名模型保存简称、旧名、英文/繁体名和用户别名。
3. 页面 autocomplete 只查询本地目录，不在每次键入时请求外部 provider。
4. 单独定义 `SecurityDirectoryProvider` 负责目录同步；定义 `SecurityMetadataEnricher` 负责已知代码后的按需校验/补全。
5. LongPort 只可作为 metadata enricher 和既有 quote-only 行情源，不能成为名称搜索的唯一目录来源。
6. 只全量维护证券元数据；报价、日 K 和分钟 K 仍按用户关注范围拉取。

## 原因

- 搜索稳定、低延迟，离线或 provider 故障时仍可用。
- 避免把输入联想变成高频外部调用，降低限流、费用和隐私风险。
- 证券标识统一后可复用于行情、板块、自选、交易和回测。
- 元数据与价格事实边界清晰，后续可替换或叠加目录来源。

## 影响

### 正面

- 搜索体验稳定，可做确定性排序、拼音和旧名检索。
- provider 变更不会扩散到所有表单组件。
- 目录同步与行情采集可独立监控、重试和验收。

### 成本与风险

- 需要维护目录同步、数据来源优先级和退市/改名治理。
- 需要为规范化名称和别名增加 migration、索引和测试。
- 数据源许可证、调用配额与再分发条款必须在接入前单独核实。

## 被否决方案

1. **输入时实时调用 LongPort 搜索**：接口能力与可用性不足以承担跨市场全目录名称检索，且搜索被网络和额度绑定。
2. **把所有证券行情全量落库再搜索**：名称搜索只需要元数据，全量行情成本与收益不匹配。
3. **为此新建 `security_master`**：与 `stock_basic` 形成双主数据源，后续关联和同步容易分叉。
4. **用大模型做名称到代码映射**：结果不可确定，可能产生不存在或错误市场的代码，不适合作为证券主键入口。

## 后续

实施边界与分期见 `docs/development/SECURITY_DIRECTORY_SEARCH_IMPLEMENTATION_PLAN.md`。任何 provider 只能提供只读元数据/行情，不改变 ADR-0006 与 ADR-0008 的安全边界。
