# 证券目录与智能检索专家评审纪要

> 日期：2026-07-17。本文记录设计推导，不代表已实现事实；最终决策以 ADR-0009 为准。

## 参与视角

- 产品与交易流程：关注用户输入、同名误选、异常状态和首批接入范围。
- 量化数据治理：关注证券身份、改名/换代码、退市判定、来源血缘和目录发布质量。
- Java/React 架构：关注现有 `stock_basic`、MyBatis 索引查询、provider 边界、共享组件和发布拆分。

## 已达成共识

1. 本地 `stock_basic` 升级为统一证券目录，不创建第二套证券主表。
2. 正常搜索只访问本地 DB；外部服务负责初始化、增量同步、本地无结果时的显式精确查找和选后校验。
3. LongPort 报价/K 线 provider 与证券目录 provider 分离；Static Info 只用于已知代码的属性补全。
4. 用户必须明确选择候选后才自动填充，名称、代码、市场来自同一条目录记录。
5. 全量保存证券元数据，行情仍按自选、持仓、计划、板块和手工任务范围获取。
6. 退市不物理删除；停牌是时点行情状态，不是证券生命周期。
7. 首批只改行情和板块四个流程，再推广到交易业务表单。

## 关键取舍

### 身份模型

短期沿用 `stock_basic.id` 作为稳定内部身份，`canonical_symbol` 作为当前业务代码；旧名称和旧 ticker 进入 alias/历史标识。发生真实 ticker 变更场景后再引入有效期化 identifier 表，避免 MVP 一次平台化过度。

### 搜索技术

当前规模先使用规范化字段、精确/前缀 B-Tree 索引和第二段包含查询，不引入 Elasticsearch、Redis 或不可解释模型排序。只有真实基准证明 MySQL 不够时才升级。

### 目录来源

CSV/官方快照是可离线验收的起点；LongPort Security List、Tushare 或其他来源是可替换实现。接入前分别核实市场覆盖、更新频率、权限、许可证与再分发约束，不能把“provider 可查询”直接等同于“权威完整名录”。

### 目录发布

全量数据先进入候选/staging，经过数量波动、唯一性、必填字段和代码映射检查再发布。一次成功响应中缺失某证券不足以判断退市，必须连续缺失并经第二来源或人工确认。

## 被否决方向

- 每次输入实时请求 LongPort 或其他 provider。
- 从报价/K 线表反推证券目录。
- 为搜索建立与 `stock_basic` 平行的主数据表。
- 用大模型直接生成证券代码。
- 为了名称搜索全量下载所有市场价格历史。
- 第一轮同时改造全部交易表单。

## 技术接手结论

按 `SECURITY_DIRECTORY_SEARCH_IMPLEMENTATION_PLAN.md` 的 D1-D4 执行。第一轮只做 D1，确保目录、导入、排名 API 和后端门禁独立可交付；不得因外部凭据或 provider 不可用中断本地功能开发。
